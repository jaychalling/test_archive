# Cluster99 앱 아이디어 필터링 룰 (Gate System) v2.0 — USA (2026)

## 목적(업데이트)

“만들기 쉬운 앱”이 아니라
① 6개월 뒤 살아남고(알고리즘·AI 검색 환경), ② 12개월 뒤 돈이 되는(AdSense), ③ 반복 확장 가능한(팩토리화) 앱만 통과.

## Gate 0 — 구조 적합성(구조 위반 즉시 탈락)

이 Gate는 “SEO 아이디어”가 아니라 “Cluster99 모델에 들어올 수 있나”를 보는 단계.
**아래 중 하나라도 ❌면 즉시 탈락:**

### 0.1 클러스터 설계가 불가능(인덱스 6–12로 닫히지 않음)

- Entry 1 + Supporting 1 + Depth 6~7 + Glue-SEO 1 구성으로 닫히지 않으면 탈락 > CLUSTER99*기술관련*검토문서
- “Depth를 5개 이상 쪼갤 수 없음” → 탈락 > 앱*아이디어*필터링*룰*(Gate_System)

### 0.2 Supporting 앵커 6–12를 만들 수 없음(변형 흡수 불가)

- 변형 검색어를 개별 페이지가 아니라 “Supporting 1페이지(앵커 6–12)”로 흡수해야 한다. 못하면 탈락.
  > CLUSTER99*기술관련*검토문서
  > 앱*아이디어*필터링*룰*(Gate_System)

### 0.3 계산기/도구가 핵심이고 문서는 보조

- 도구 중심은 Glue-UX로 격리(noindex)하는 게 원칙.
- “도구가 핵심이고 문서가 뒤에 붙는 구조”면 Cluster99 철학과 정면충돌 → 탈락
  > 앱*아이디어*필터링*룰*(Gate*System)
  > CLUSTER99*기술관련\_검토문서

### 0.4 Combination(조합형 개별 URL) 유혹이 큰 주제

- “금액×연도×지역” 같은 조합형 URL을 만들게 되는 주제는 구조적으로 near-duplicate 리스크가 커서 원칙적으로 탈락
  > CLUSTER99*기술관련*검토문서

**✅ 통과 예(구조가 닫힘)**

- “How much was X worth historically?” → Supporting 앵커(연도/카테고리) + Depth(큰 축)로 닫힘 > 앱*아이디어*필터링*룰*(Gate_System)

**❌ 탈락 예**

- “X calculator”, “X meaning”, “X definition” (Depth/Supporting이 잘 안 나옴) > 앱*아이디어*필터링*룰*(Gate_System)

## Gate 1 — 검색 의도 밀도(연쇄 질문 생성력)

Helpful/People-first 환경에서는 “한 번 보고 끝”나는 페이지가 너무 위험.
**통과 기준: 아래 질문이 5개 이상 자동으로 따라 나와야 함(최소 3개면 보류, 2개 이하면 탈락)**

- “평균/중앙값은?”
- “상황/지역/연령/직업별로 다르지 않나?”
- “지금 기준으로 어느 수준?”
- “비슷한 사례/대체 옵션은?”
- “장기 추세는?”
- “사람들이 흔히 착각하는 포인트는?”
- “의사결정(돈/시간)을 바꿀 만큼 유용한가?”

👉 여기서 핵심은 Depth 6~7개가 ‘자연스럽게’ 생기냐다. “억지 분화”면 다음 Gate에서 터진다.

## Gate 2 — E-E-A-T(특히 Trust) 구현 가능성

미국 검색은 이제 “정보가 맞냐/믿을 수 있냐/누가 책임지냐”가 핵심이다.
**다음 중 3개 이상 충족해야 통과(2개면 보류, 1개 이하면 탈락):**

### 2.1 “검증 가능한 데이터 소스”가 존재

- 정부/공공기관/대형 리서치/권위 있는 통계 등
- 최소 2개 이상 서로 다른 출처로 크로스체크 가능

### 2.2 “방법론/가정”을 문서로 설명 가능

- 계산 기준, 인플레이션 기준년도, 세율 적용 로직 등
- “왜 이렇게 계산했는지”를 문서화할 수 있어야 함(Depth에서 신뢰 형성)

### 2.3 “최신성 유지”가 가능한 주제

- 업데이트 빈도(월/분기/연)와 데이터 수급 경로가 명확해야 함
- “연도만 바꾸면 되는 템플릿”이 아니라, 업데이트 스토리가 있어야 함(신뢰 신호)

### 2.4 YMYL(돈/건강/법)일수록 더 강한 Trust 설계 가능

- 세금/은퇴/금융은 CPC 좋지만, 근거/면책/출처/업데이트 없으면 오히려 위험

**탈락 패턴(현실적으로 많이 망함)**

- “출처 없이 ‘그럴듯한 이야기’만 가능”
- “업데이트 필요하지만 데이터가 없다”
- “책임 회피형 문장만 반복(Trust 하락)”

## Gate 2-A — E-E-A-T 최소 구현 체크리스트 (실행 규칙)

Gate 2는 “가능한가?”를 묻는 필터였다.
Gate 2-A는 **“반드시 이렇게 구현한다”는 실행 규칙**이다.

이 Gate를 통과하지 못하면:

- E-E-A-T는 *의도*만 있고
- 실제 페이지 간 품질 편차가 발생하며
- 도메인/섹션 단위 신뢰 신호가 깨질 위험이 커진다.

따라서 **Gate 2를 통과한 모든 앱은 Gate 2-A 구현을 의무 적용**한다.

### 2-A.1 Entry 페이지 필수 구성 (하단 고정 영역)

Entry 페이지에는 반드시 아래 3개 블록을 포함해야 한다.

#### (1) Data Sources

- **최소 2개 이상**
- 서로 다른 성격의 출처 권장
  - 예: 정부 통계 + 대형 리서치
- 링크는 외부 링크로 명시

**표준 형식 예시**

- U.S. Bureau of Labor Statistics (CPI-U, 1970–2025)
- Federal Reserve Economic Data (FRED)
- IRS / SSA / Census Bureau 등

> ❌ “일반 블로그 인용”, “출처 없는 요약”은 인정하지 않음

#### (2) Methodology (요약 5줄 내외)

- “무엇을 기준으로 계산했는가”
- “어떤 가정을 두었는가”
- “왜 이 방식이 합리적인가”

**규칙**

- 최소 3줄, 최대 7줄
- 수식 전체를 나열하지 않아도 되지만 **논리 흐름은 반드시 설명**
- 마케팅 문장 금지

#### (3) Last Updated

- 날짜 필수 명시 (YYYY-MM-DD)
- 자동 업데이트가 아니어도 됨
- “Updated when new data is available” 같은 추상 문구 금지

### 2-A.2 Depth 페이지 필수 구성 (문서 신뢰 박스)

모든 Depth 페이지에는 아래 중 **최소 1개 이상**을 반드시 포함한다.

#### 옵션 A — Calculation / Formula Box

- 실제 계산식
- 단계별 설명
- 기준 연도 / 기준 수치 명시

#### 옵션 B — Assumptions Box

- 시나리오 가정
- 평균 vs 중앙값 구분
- 예외 조건 명시

> ❌ Depth가 단순 서술로만 끝나면 Trust 축적 실패로 간주

### 2-A.3 업데이트 책임 명시 (권장)

가능한 경우, 다음을 명시한다.

- “This page is maintained by Cluster99”
- 데이터 업데이트 주기 (연 1회 / 분기 / 수시 등)

이는 **YMYL 주제에서 신뢰 신호 강화**에 도움을 준다.

## Gate 3 — Thin/near-duplicate 위험도 평가(구조적으로 예방 가능해야 함)

Helpful 시스템이 코어에 녹아 들어간 이후, 얕은/중복 콘텐츠는 섹션 전체를 끌어내릴 수 있다는 우려가 계속 강조된다.
Cluster99는 이를 “인덱스 절대량 + 역할 분리 + Supporting 흡수 + Depth 구조 다양화”로 막는 모델이다. > CLUSTER99*기술관련*검토문서

**다음 중 하나라도 해당하면 탈락 또는 강한 제약(아이디어 수정 요구):**

### 3.1 “페이지를 늘릴수록 같은 말 반복”이 되는 주제

- 예: 주/도시/직업/연도 조합을 나열하면 문장 구조가 동일해짐
- 해결책이 “페이지 증가”밖에 없다면 탈락

### 3.2 Depth 6개가 ‘각각 다른 문서’로 보이게 만들 수 없음

- 각 Depth가 표/구조/해석 포인트가 달라야 함(near-duplicate 방지) > CLUSTER99*기술관련*검토문서
- “섹션 배열이 거의 동일”해질 게 보이면 탈락

### 3.3 Supporting을 Glue-UX처럼 만들게 되는 주제

- Supporting은 문서형 앵커 가이드여야 한다.
- 변형을 “계산 UI”로 처리하려 하면 Glue-UX가 되어 noindex로 떨어지고, 변형 검색을 못 먹음 → 구조적으로 손해 > CLUSTER99*기술관련*검토문서

## Gate 4 — AI Overviews/제로클릭 환경 적합성(GEO)

2025년 이후 AI 요약이 늘면서 **클릭 없이 답을 얻는 검색(제로클릭)**이 늘고 있다.
그럼에도 “출처로 인용되는 페이지”는 남는다. 즉 목표가 “순위”만이 아니라 “인용될 만한 문서”여야 한다.

**다음 4개 중 3개 이상 가능해야 통과:**

### 4.1 “한 문장 Quick Answer”를 낼 수 있음

- Entry 상단 1스크린에서 답이 끝나야 함(Cluster99 Entry 규정) > CLUSTER99*기술관련*검토문서

### 4.2 “후속 질문을 한 페이지에서 다 커버” 가능

- Supporting 앵커 6–12 + Depth 6–7로 질문군을 닫을 수 있어야 함 > CLUSTER99*기술관련*검토문서

### 4.3 “명시적 근거(출처/수치/방법)”를 제공 가능

- AI 요약은 ‘근거 있는 출처’를 선호하는 방향으로 가고 있음(적어도 사람/엔진 모두에게 신뢰 신호가 필요)

### 4.4 “FAQ 스타일(질문 헤딩)”로 문서 구조화 가능

- Q&A 섹션을 자연스럽게 넣을 수 있어야 함(리치 결과/이해도 상승)

**탈락 예**

- 답이 단일 정의로 끝나며 추가 질문이 없음
- 데이터 근거가 약하거나 변동이 심해 신뢰를 구축하기 어려움

## Gate 5 — 광고 적합성(RPM/CPC) + 페이지 체류 설계

Cluster99는 AdSense only. 그러면 “트래픽”보다 RPM이 더 중요한 순간이 많다.

> Cluster99사업계획서
> 앱*아이디어*필터링*룰*(Gate_System)

**통과 조건(모두 충족 권장, 최소 2개 충족)**

### 5.1 광고 카테고리 적합

- Finance / Tax / Insurance / Housing / Education 등(높은 CPC 기대) > 앱*아이디어*필터링*룰*(Gate_System)

### 5.2 Depth가 “긴 체류”를 만들 구조인지

- 표/비교/시나리오/FAQ로 2,000자+ 문서가 자연스럽게 나오는 주제 > 앱*아이디어*필터링*룰*(Gate_System)

### 5.3 “페이지/세션”이 설계되는 주제인지

- Entry → Depth 2개 이상 클릭이 자연스러운 흐름인지(내부 전환) > CLUSTER99*기술관련*검토문서

**탈락 예**

- SNS성 테스트/오락(AdSense RPM 낮고 변동 큼) > 앱*아이디어*필터링*룰*(Gate_System)
- 위키형 정의(체류/전환이 약함) > 앱*아이디어*필터링*룰*(Gate_System)

## Gate 6 — 데이터 확보/유지 가능성(운영 현실 필터)

2025~2026 검색은 “업데이트/정확성/책임”이 점점 중요해진다. 데이터가 없으면 E-E-A-T를 못 만든다.

**통과 조건(3개 이상):**

- 신뢰 가능한 1차/2차 데이터 소스가 존재
- 업데이트 주기(월/분기/연)가 정의 가능
- 자동/반자동 업데이트 루틴 설계 가능(수작업 지옥 금지)
- 데이터가 바뀌어도 “설명 문서(Depth)”가 의미를 유지
- 잘못된 데이터가 들어왔을 때 롤백/정정 프로세스가 존재

## Gate 7 — 기술 구현 난이도(Cluster99 팩토리와의 합치)

“만들 수 있냐”는 당연히 중요하지만, 여기서의 기준은 “Cluster99 생산 라인에서 반복 생산 가능한가”다.

### 7.1 필수: 템플릿 + JSON으로 구현 가능

- Entry/Supporting/Depth 템플릿 + JSON 데이터 조합으로 생산성 유지 > CLUSTER99*기술관련*검토문서

### 7.2 금지: 앱마다 완전히 다른 복잡한 백엔드 필요

- DB/로그인/결제 등은 지금 모델과 충돌(속도 저하)
- 필요한 경우에도 “noindex 도구(Glue-UX)”로 격리하고 SEO는 문서가 담당 > CLUSTER99*기술관련*검토문서

## Gate 8 — 섹션 품질 리스크(도메인 전체 보호)

구글은 사이트/섹션의 품질 신호를 본다는 논의가 계속되고, “저품질이 섞이면 전체가 손해”가 되는 사례가 많다. (이건 운영 관점에서 무조건 보수적으로 가야 한다.)

**다음 중 하나라도 해당하면 격리(별도 도메인/서브디렉토리 분리) 또는 탈락:**

- 오락/루머/검증 어려운 건강 주장
- 민감 주제(건강·법률)인데 출처/면책/업데이트 체계 없음
- UGC/자동 생성 문구가 메인이 되는 구조

Cluster99는 “클러스터(앱) 단위 운영”이므로, 실험은 앱 단위로 격리하고 noindex-first로 안전하게 진행해야 한다는 원칙과 연결된다. > CLUSTER99*기술관련*검토문서

## Gate 8-A — 실험용 아이디어 Sandbox 규칙 (Labs)

Gate 8은 “도메인 보호”를 위한 보수적 필터다.
하지만 **모든 아이디어를 통과/탈락으로만 나누면 잠재적 히트를 잃을 수 있다.**
이를 위해 **Sandbox(Labs) 규칙을 추가한다.**

### 8-A.1 Sandbox 대상 정의

다음 조건을 만족하면 Sandbox 실험이 가능하다.

- Gate 0 통과
- Gate 1~7 중 일부 미달 (총점 25~34점)
- 구조적 위험은 낮으나
  - 검색 의도 밀도가 애매하거나
  - 수요 검증이 불충분한 경우

### 8-A.2 Sandbox 구현 규칙 (필수)

Sandbox 앱은 다음 규칙을 따른다.

- URL: `/labs/{app-name}/`
- **noindex, follow** 기본 적용
- 내부 링크:
  - Labs 내부에서만 순환
  - 메인 섹션으로의 링크 금지
- 광고:
  - 최소화 또는 비활성화 가능
    > 목적은 **SEO 성과가 아니라 신호 탐지**다.

### 8-A.3 실험 기간 및 평가

- 실험 기간: **30~60일**
- 평가 기준 (Gate 9 일부 차용)

**필수 관찰 지표:**

- 자연 검색 Impression 발생 여부
- Avg engagement time
- Entry → Depth 클릭 발생 여부

### 8-A.4 정식 편입 조건

Sandbox 앱이 아래 중 **2개 이상 충족 시** 정식 편입 검토 가능:

- Search Console에서 유의미한 Impression 증가 추세
- Depth 체류시간이 메인 앱 평균 이상
- 후속 질문(Depth 확장) 설계 가능성 확인

**편입 시:**

- `/labs/` → 메인 카테고리로 이동
- index 허용
- Gate 2-A(E-E-A-T) 전면 적용

### 8-A.5 실패 처리 원칙

- 실험 종료 후 성과 없을 경우:
  - 삭제 또는 영구 noindex 유지
- 실패 Sandbox는 **도메인 품질 점수에 영향을 주지 않음**

## 최종 통과 판정 방식(점수제, 실제로 굴릴 수 있게)

각 Gate를 점수화해서 “감”으로 통과시키지 않게 만든다.

**점수표(권장)**

- Gate 0: Pass/Fail (Fail이면 즉시 탈락)
- Gate 1~9: 각 0~5점
- **총점 35점 이상 + Gate2(E-E-A-T) 3점 이상 + Gate3(중복리스크) 3점 이상만 통과**

**Gate System 요약 업데이트**

- Gate 2 → “가능성 평가”
- **Gate 2-A → E-E-A-T 실행 표준**
- Gate 8 → 도메인 보호
- **Gate 8-A → 실험용 안전 구역**

이로써 Cluster99는 **보수적 보호 + 공격적 실험을 동시에 수행하는 구조**를 갖는다.

## 예시로 보는 “통과/탈락” 패턴

**✅ 통과 패턴 A (Cluster99에 가장 잘 맞음)**

- FAQ 구조화 쉬움
- 출처 명확히 달 수 있음(Trust)
